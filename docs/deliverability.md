# Deliverability playbook

The existential constraint for BetterSite is email deliverability. A burned
sender reputation ends the POC. This document is the runbook for keeping
the inboxes healthy.

## The 14-day rule

**Sender domains must start their warm-up 14 days before the first real
cold send.** This is the longest-lead-time item in the entire POC schedule.

Warm-up happens in parallel with engineering (Track B in `PROGRESS.md`). It
is not optional and it cannot be compressed.

## Step 1 — Buy domains

Target: 2-3 sender domains, distinct from `bettersite.co` (the product
domain). Suggestions:

- `betterstudio.co`
- `bettersites.co`
- `bettersitepro.com`

Criteria:
- `.com` / `.co` / `.net` — avoid novelty TLDs, they're flagged more often
- Registered at least 30 days ago (newly-registered domains are penalized)
- No prior spam history (check MXToolbox, SORBS)

## Step 2 — Configure SPF / DKIM / DMARC

For every sender domain, before any email is sent:

**SPF** (TXT record at the domain apex):
```
v=spf1 include:<your-sending-provider> -all
```

**DKIM** — generated by the sending provider, set as a TXT record at the
provider-specified selector.

**DMARC** (TXT record at `_dmarc.<domain>`):
```
v=DMARC1; p=none; rua=mailto:dmarc-reports@<domain>; fo=1
```

Start at `p=none` for the first month, move to `p=quarantine` once
deliverability is stable.

Verify with MXToolbox or `dig` before proceeding.

## Step 3 — Set up inboxes

Target: 2-3 inboxes per domain → 4-9 inboxes total.

Common names that don't look suspicious:
- `alex@`, `sarah@`, `founder@`, `hello@`

Avoid: `no-reply@`, `info@`, `contact@` — these get filtered harder.

## Step 4 — Warm up

Use a warmup service (Mailwarm, Warmup Inbox, or Instantly's built-in
warmup). Configure each inbox to:

- Start at ~20 emails/day outbound
- Ramp by ~5 emails/day for 14 days → ~90/day at steady state
- Maintain a conversation-like exchange pattern (the warmup service
  simulates this automatically)

Do NOT send any real cold emails during warmup. Any early send damages
the reputation curve and restarts the clock.

## Step 5 — Pre-launch seed test

Before sending the first real batch:

1. Send the actual cold email template (with a real preview link pointed
   at a real test lead) to a set of inboxes you control:
   - At least 3 Gmail addresses
   - At least 2 Outlook addresses
   - At least 1 Yahoo address
   - At least 1 Apple Mail address
   - At least 1 ProtonMail address
2. Wait 10 minutes.
3. Check each inbox manually. Every single one must land in **Primary /
   Inbox** — not Promotions, not Updates, not Spam.
4. If any land in the wrong folder, stop. Investigate the specific
   provider's rejection reason before touching the batch.

See `scripts/seed_test_email.py`.

## Step 6 — First batch

- Start with 100 emails.
- Distribute across all inboxes (not all from one domain).
- Monitor bounce rate + complaint rate hourly for the first 6 hours.
- If bounce rate exceeds 2% OR complaint rate exceeds 0.1% on any inbox,
  halt that inbox immediately and investigate.

## Daily monitoring (first week)

Check per-inbox:
- Bounce rate (target: < 2%)
- Complaint rate (target: < 0.1%)
- Open rate (healthy baseline: 30-50%)

The admin dashboard's deliverability view (Phase 5) shows this
automatically. Until then, pull numbers manually from the sending
provider's dashboard.

## Common failure modes

- **Brand-new domain sending cold email:** reputation = 0 → Gmail spam.
  Always warm up first.
- **Wrong SPF/DKIM:** silent landing in spam. Re-verify after any DNS
  change.
- **Image-heavy email with one link:** spam filters hate this pattern.
  Balance with substantive text content. This is why the embedded
  preview screenshot (cherry-pick #1) needs its own deliverability test
  before launch.
- **Sending too fast:** even warmed inboxes get flagged at >200/day.
  Keep the rotation slow.
- **Shared IP pool:** if the sending provider uses shared IPs, other
  users' bad behavior can tank your deliverability. Prefer dedicated IPs
  when available.
